# Database Schema

## Context Status

This schema is generated from the Step 1 artifacts, not from a formal product specification. It is intentionally compact and optimized for prototype evaluation. Names use `snake_case` for database objects and `camelCase` for API fields.

## Assumptions

- PostgreSQL is the default relational database for this specification.
- IDs are UUIDs generated by the backend or database.
- Roles are household-scoped through `household_memberships`.
- Sensitive child and household information is protected through application-level authorization.
- Payment, marketplace, community, and professional-provider features are out of scope for this prototype.

## Core Tables

### users

Stores authenticated app users.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| id | uuid | yes | primary key |
| name | text | yes | display name |
| email | text | no | unique when present |
| phone | text | no | unique when present |
| auth_provider | text | yes | example: `email`, `google`, `apple` |
| status | text | yes | `active`, `disabled` |
| created_at | timestamptz | yes | creation timestamp |
| updated_at | timestamptz | yes | update timestamp |

Constraints:

- at least one of `email` or `phone` should be present
- unique index on `email`
- unique index on `phone`

Connects users to households with scoped roles.

| household_id | uuid | yes | references `households.id` |

## Deferred Tables

These are intentionally not included in the prototype schema:

- subscriptions
- payments
- marketplace listings
- community groups
- professional providers
- audit logs

Audit logs may be added later if privacy or compliance requirements become explicit.
